# Task 拆解与编写规范

从技术方案（04）出发，以 03 功能清单为唯一依据，将功能拆解为可独立开发、可独立测试、可独立评审的原子级 Task。

## 适用场景

- 研发流程 Phase 2 Step 6：技术方案已产出，需要拆解 Task 清单
- 独立进行 Task 拆解（用户直接要求「拆 Task」「出任务清单」）

---

## 硬约束

- Task 拆解的**唯一依据**是 03 功能清单，不得自行发挥功能范围。
- 03 中每条 P0 功能项至少映射 1 个 Task，不得遗漏。
- 每个 Task 必须关联至少 1 个测试用例 ID（TC 编号）。
- 单个 Task 修改文件 ≤ 5 个、预估工时 ≤ 2h；超出则继续拆分。
- 同一功能的前端和后端实现必须拆为不同 Task。
- 依赖关系不得存在循环。

## 拆解原则

### 1. 功能全覆盖

03 功能清单是拆解的唯一输入。操作步骤：

1. 列出 03 中所有 F 编号
2. 对照 04 技术方案映射表，确认每条 F 的技术实现要点
3. 将实现要点拆为具体 Task
4. 填写功能覆盖矩阵，确认无遗漏

### 2. 前后端分层

同一功能的前端和后端拆为不同 Task：

| 功能 | 后端 Task | 前端 Task |
|------|----------|----------|
| F01 搜索 | TASK-001 搜索接口实现 | TASK-002 搜索页面实现 |
| F02 详情 | TASK-003 详情接口实现 | TASK-004 详情页面实现 |

好处：
- 前后端可并行开发（后端先出接口，前端先用 Mock）
- 测试粒度更细（后端接口测试 + 前端 E2E 分开验证）
- 评审范围可控

### 3. 原子性检验

每个 Task 必须通过以下检验：

| 检验项 | 标准 | 不通过时 |
|--------|------|---------|
| 修改文件数 | ≤ 5 个 | 继续拆分 |
| 预估工时 | ≤ 2h | 继续拆分 |
| 可独立测试 | 有关联 TC 可验证 | 补充 TC 或重新划分边界 |
| 可独立评审 | 不依赖同批 Task 的代码改动 | 调整依赖关系 |

### 4. 依赖显式化

- 有前置依赖的 Task 必须在元数据中标注依赖项编号
- 依赖关系必须在索引文件的依赖关系图中可视化
- 禁止循环依赖

判断依赖的方法：
- Task B 用到 Task A 创建的接口/组件/类型 → B 依赖 A
- Task B 修改 Task A 同一文件的不同部分 → 考虑合并或明确串行

### 5. 测试全关联

每个 Task 至少关联 1 个测试用例（TC 编号）：

- TC 编号来源 1：03 功能清单中的验收标准 TC
- TC 编号来源 2：06 测试用例中研发补充的技术 TC（`TC-XXX-NNN-U` / `TC-XXX-NNN-I`）

Task 完成的定义 = 关联的 TC 全部通过 + 覆盖率达标。

## 常见拆分模式

### 模式 A：CRUD 功能

```text
F01 管理功能 →
  TASK-001 后端：数据模型与 CRUD 接口
  TASK-002 前端：列表页
  TASK-003 前端：新增/编辑表单
  TASK-004 前端：删除与确认交互
```

### 模式 B：搜索/筛选功能

```text
F02 搜索功能 →
  TASK-005 后端：搜索接口（含分页、排序、筛选）
  TASK-006 前端：搜索栏与筛选面板
  TASK-007 前端：搜索结果列表与分页
```

### 模式 C：第三方集成

```text
F03 第三方集成 →
  TASK-008 后端：第三方 API 对接与适配层
  TASK-009 后端：数据同步/回调处理
  TASK-010 前端：集成状态展示与错误处理
```

### 模式 D：基础设施（跨功能依赖）

```text
基础设施 →
  TASK-000 项目初始化（脚手架、数据库迁移、基础配置）
  → 被多个业务 Task 依赖
```

## Task 清单自检（产出后必执行）

| 自检项 | 检查方法 | 通过标准 |
|--------|---------|---------|
| 功能全覆盖 | 对照 03 F 编号与功能覆盖矩阵 | 每条 P0 F 至少被 1 个 Task 覆盖 |
| 测试全关联 | 检查每个 Task 的关联 TC 字段 | 每个 Task 至少关联 1 个 TC |
| 依赖无环 | 按依赖关系图走一遍 | 无法回到起点 |
| 工时合理 | 检查总览表工时列 | 单 Task ≤ 2h |
| 文件数可控 | 检查每个 Task 的修改文件清单 | 单 Task ≤ 5 个文件 |
| ID 连续 | 检查 Task ID 序号 | 无跳号、无重复 |
| 层级标注 | 检查总览表层级列 | 每个 Task 标注前端/后端/全栈 |

---

## 常见反模式

- 一个 Task 同时包含前后端实现，导致修改文件 > 5 个且无法独立评审
- Task 描述只写「实现 F01」，缺少具体实现要点和修改文件清单
- 依赖关系靠猜测而不是显式标注，开发时才发现阻塞
- 所有 Task 串行排列，没有识别可并行的 Task
- 关联 TC 字段为空，Task 完成后无法验证
- 将「修复 Bug」「代码审查」作为独立 Task（这些是流程活动，不是交付物）
- 基础设施 Task 未从业务 Task 中分离，导致多个 Task 重复做环境准备
