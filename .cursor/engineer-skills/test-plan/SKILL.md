# 测试规划规范

从 03 功能清单的验收标准（TC）出发，产出完整的测试用例文档和可执行的测试脚本，覆盖产品验收层和技术补充层。

## 适用场景

- 研发流程 Phase 3 Step 7：Task 清单已产出，需要编写测试用例和测试脚本
- 独立编写测试计划（用户直接要求「写测试用例」「出测试方案」）
- TDD 模式下先于 Task 拆解产出测试脚本

---

## 硬约束

- 03 中的每条 TC 必须在测试用例文档中逐条出现，保持 TC ID 不变。
- 每个 API 端点至少有 1 个正常用例 + 1 个异常用例。
- 前端 E2E 必须覆盖 03 中全部 P0 功能项的关键路径。
- 测试脚本必须可直接执行（`npm test` / `npx playwright test`），不可只有文档没有脚本。
- 覆盖率要求：后端核心业务逻辑行覆盖率 ≥ 80%，前端组件行覆盖率 ≥ 60%。

## 后端集成测试脚本规范

### 用例结构（参考 IT-xx 模式）

每个测试文件按功能域组织，每个用例遵循以下结构：

```typescript
describe('<功能域名称>', () => {
  // 前置数据准备
  beforeAll(async () => {
    // 插入测试数据、初始化 Mock
  });

  // 清理
  afterAll(async () => {
    // 清理测试数据
  });

  describe('<API 端点>', () => {
    // IT-xx: 场景描述
    it('IT-01: 正常请求返回预期结果', async () => {
      // Arrange: 准备请求参数
      // Act: 发送请求
      // Assert: 断言响应状态、数据结构、关键字段
    });

    it('IT-02: 缺少必填参数返回 400', async () => {
      // Arrange → Act → Assert
    });
  });
});
```

### 编写规则

| 规则 | 说明 |
|------|------|
| 用例命名 | `IT-NN: 场景简述`，与测试用例文档中的 TC ID 对应 |
| 三段式 | 每个用例包含 Arrange / Act / Assert 三段 |
| 数据隔离 | 每个用例使用独立业务 key 或独立数据，断言后清理 |
| Mock 策略 | 外部依赖（Feign/HTTP）使用 Mock；数据库使用测试库 |
| 断言粒度 | 断言响应状态码 + 关键字段值，不只断言"不报错" |

### 冒烟用例（Smoke）

冒烟用例串联多个接口，验证端到端业务流程：

```typescript
describe('Smoke: <业务链路名称>', () => {
  it('SM-01: 完整业务流程', async () => {
    // Step 1: 调用 API-A
    const resultA = await request.post('/api/a', dataA);
    expect(resultA.status).toBe(200);

    // Step 2: 用 API-A 返回值调用 API-B
    const resultB = await request.get(`/api/b/${resultA.data.id}`);
    expect(resultB.status).toBe(200);

    // Step 3: 验证最终状态
    expect(resultB.data.status).toBe('completed');
  });
});
```

### 后端集成测试清单

每个 API 端点必须覆盖：

| 场景类型 | 必须/可选 | 示例 |
|---------|----------|------|
| Happy Path | **必须** | 正常参数，返回 200 + 预期数据 |
| 参数缺失 | **必须** | 缺少必填字段，返回 4xx |
| 非法参数值 | **必须** | 参数超范围/类型错误，返回 4xx |
| 资源不存在 | 推荐 | 查询/修改不存在的 ID，返回 404 或业务失败 |
| 并发冲突 | 按需 | 乐观锁/幂等场景 |
| 权限不足 | 按需 | 无权限用户访问 |

## 前端 E2E 测试脚本规范

### 技术栈

- 测试框架：**Playwright**
- 脚本格式：`.spec.ts` 或 `.spec.mjs`
- 参考模式：`原型/模块/<模块>/验收/acceptance.spec.mjs`

### 用例结构

```typescript
import { test, expect } from '@playwright/test';

test.describe('<功能域>', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/target-page');
    // 等待页面加载完成
    await page.waitForSelector('[data-testid="page-loaded"]');
  });

  test('TC-XXX-001: <场景描述>', async ({ page }) => {
    // 操作步骤
    await page.click('[data-testid="search-btn"]');
    await page.fill('[data-testid="search-input"]', '搜索词');
    await page.click('[data-testid="submit-btn"]');

    // 断言
    await expect(page.locator('[data-testid="result-list"]')).toBeVisible();
    await expect(page.locator('[data-testid="result-item"]')).toHaveCount(3);
  });
});
```

### 必须覆盖的场景

| 场景类型 | 说明 | 示例 |
|---------|------|------|
| 主路径（Happy Path） | 核心用户操作流程完整走通 | 搜索 → 查看结果 → 进入详情 |
| 关键分支 | 异常输入、空状态、错误降级 | 搜索无结果 → 展示空状态引导 |
| 交互状态 | 加载态、流式输出、响应式切换 | 提交后展示 loading → 成功提示 |
| 移动端适配 | 关键页面在移动端视口下可用 | 768px 宽度下侧栏折叠、内容不溢出 |

### 选择器策略

优先级从高到低：

1. `data-testid` 属性（最稳定）
2. ARIA role + name（语义化）
3. 文本内容（适合按钮、链接）
4. CSS 选择器（最后手段，容易因样式重构失效）

### 截图与证据

```typescript

await page.screenshot({ path: `screenshots/TC-XXX-001-step1.png` });

// 失败自动截图（Playwright 配置）
// playwright.config.ts 中设置：
```

## 测试分层策略

```text
┌─────────────────────────────────────────┐
│        E2E 测试（Playwright）            │  ← 覆盖 P0 关键路径
│  验证用户视角的完整业务流程               │     数量少、粒度粗
├─────────────────────────────────────────┤
│        集成测试（Integration）            │  ← 覆盖全部 API 端点
│  验证接口行为、数据流转、外部依赖交互      │     数量中、粒度中
├─────────────────────────────────────────┤
│        单元测试（Unit）                   │  ← 覆盖核心业务逻辑
│  验证纯函数、工具方法、数据转换            │     数量多、粒度细
└─────────────────────────────────────────┘

执行速度：单元 > 集成 > E2E
覆盖信心：E2E > 集成 > 单元
```

### 各层职责

| 层级 | 测试对象 | Mock 范围 | 执行频率 |
|------|---------|----------|---------|
| 单元测试 | 纯函数、工具方法、数据转换 | 所有外部依赖 | 每次提交 |
| 集成测试 | API 端点、服务层、数据访问层 | 外部 HTTP 调用；数据库用测试库 | 每个 Task 完成 |
| E2E 测试 | 完整用户操作流程 | 后端可选 Mock 或真实服务 | 验收阶段 |

## 常见反模式

- TC 只写「功能正常」而非具体的操作步骤和预期结果
- 后端测试只测 Happy Path，不测异常/边界场景
- E2E 测试依赖 CSS class 选择器，样式重构后全面失效
- 测试数据硬编码在脚本中，无法适应不同环境
- 冒烟用例与单接口用例重复，未体现端到端串联价值
- 只有测试文档没有可执行脚本，或只有脚本没有文档
- Mock 数据与真实数据结构不一致，测试通过但实际集成失败
- 覆盖率只看行数不看分支，遗漏重要条件分支
