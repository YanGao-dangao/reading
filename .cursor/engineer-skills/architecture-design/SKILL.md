# Architecture Design

用于前端架构设计、结构调整与模块规划。目标是让页面、组件、Hook、API、状态与工具代码职责清晰、依赖单向、演进可控。

> **场景导航**
> - 新建页面或组件 → `/build-page`
> - 落地 React 实现 → `/react-guardrails`
> - 重构既有代码 → `/code-refactoring`
> - 编码风格规范 → `/coding-standards`

## 硬约束

- 架构调整默认是**等价迁移**，不得顺带改变既有业务语义。
- 先定义边界，再落代码；不得一边实现一边随意漂移职责。
- 依赖方向必须稳定、单向，禁止循环依赖与跨层反向依赖。
- 公共层只放稳定复用能力，禁止把"当前页面特有逻辑"提前抽到全局。
- 共享组件、共享 Hook、共享工具不得直接依赖页面文件。
- `api` 层只处理数据访问与接口适配，不承载页面交互逻辑。
- `store` 只保存跨页面或跨模块共享状态，页面私有状态优先本地管理。
- 大段业务逻辑（校验、计算、转换）应从组件中抽离到独立的 Hook 或 utils 文件，组件内允许保留少量简单内联逻辑。

## 设计目标

1. **高内聚**：同一业务变化尽量集中在少量文件内。
2. **低耦合**：页面、业务模块、基础设施之间边界明确。
3. **易扩展**：新增场景以增加模块为主，而不是持续修改旧入口。
4. **易回收**：页面下线、需求回退时可以按模块删除，不牵连全局。

## 推荐分层

按"离用户界面多近、离业务规则多近、离基础能力多近"进行分层。

```text
┌─────────────────────────────────────────────────────┐
│  1. 页面编排层 (Page Orchestration)                   │
├─────────────────────────────────────────────────────┤
│  2. 业务视图层 (Domain UI)                            │
├─────────────────────────────────────────────────────┤
│  3. 业务逻辑层 (Domain Logic)                         │
├─────────────────────────────────────────────────────┤
│  4. 数据访问层 (Data Access)                          │
├─────────────────────────────────────────────────────┤
│  5. 状态共享层 (Shared State)                         │
├─────────────────────────────────────────────────────┤
│  6. 共享基础层 (Shared Infrastructure)                │
└─────────────────────────────────────────────────────┘

依赖方向（向下为主）:
Page / Domain UI ──读取──→ Store
Page → Domain UI → Domain Logic → Data Access → Shared Infra / Types
Store → Data Access → Shared Infra / Types
```

### 1. 页面编排层

放在 `src/pages/` 或页面目录下的入口文件中。

**职责：**
- 路由入口、页面级布局、页面流程编排
- 组合页面级组件 / Hook / API
- 处理页面级 loading、empty、error 态
- 处理页面导航、弹窗开关、提交入口等流程控制

**禁止：**
- 写大量可复用业务规则
- 堆积复杂数据转换
- 直接承担跨页面共享逻辑

### 2. 业务视图层

放在页面目录下的 `components/`。

**职责：**
- 当前业务域的 UI 组件（弹窗、卡片、表单片段、列表片段）
- 组合业务逻辑层提供的 Hook 与数据，完成渲染
- 对页面层提供语义化组件接口，隐藏内部 UI 细节

**禁止：**
- 在组件中内联大段业务计算、校验、数据转换逻辑
- 在组件中内联复杂请求编排（多接口串联、提交 + 刷新 + 提示等流程），简单的单接口查询允许组件直接调用

### 3. 业务逻辑层

放在页面目录下的 `hooks/`、`utils/`。

**职责：**
- 业务 Hook：请求编排、提交流程、草稿逻辑、状态管理
- 业务纯函数：校验规则、得分计算、状态映射、数据格式化
- 业务级数据适配器（Mapper）：复杂业务转换，如根据多字段派生状态、将列表扁平化为树等
- 对业务视图层提供语义化 API，而不是暴露细碎实现细节

**特征：**
- `utils/` 中的函数应尽量保持纯函数，可独立单测，不依赖 React 运行时
- `hooks/` 中的 Hook 可依赖 React，但不依赖具体组件

**数据适配器命名约定：**
- 统一放在页面 `utils/` 或 `api/` 同目录
- 命名建议：`xxxMapper.ts` 或 `xxxAdapter.ts`

**适合上提到共享的条件：**
- 已在 2 个及以上页面复用
- 语义稳定，不依赖某一个页面上下文
- 抽出后不会引入一堆条件分支

### 4. 数据访问层

**全局 API** 放在 `src/api/`，**页面私有 API** 放在 `pages/xxx/api/`。

**职责：**
- 请求发送、响应类型声明、接口参数适配
- 接口级数据整理（字段重命名、null 兜底、日期格式标准化等），确保不向上层暴露原始后端结构
- 屏蔽接口差异，向上层暴露稳定数据结构

**两级转换策略：**

| 转换类型 | 归属层 | 举例 |
|----------|--------|------|
| 接口适配（字段重命名、null 兜底、日期格式化） | 数据访问层（API 内部） | `createTime` → `createdAt`，`null` → `''` |
| 业务转换（派生字段、聚合、状态映射） | 业务逻辑层（Mapper / utils） | 根据多字段计算得分等级、将列表扁平化为树 |

**API 放置策略：**

| 位置 | 适用场景 |
|------|----------|
| `src/api/` | 跨页面共用、或已稳定的基础接口 |
| `pages/xxx/api/` | 仅当前页面域使用、尚未跨域复用的接口 |

**演进规则：** 当页面级 API 被第二个页面域引用时，应上提到 `src/api/`。

**禁止：**
- 操作 DOM 或弹 Toast
- 依赖页面组件
- 在接口函数中混入页面流程判断
- 直接返回未经整理的原始后端数据结构

### 5. 状态共享层

放在 `src/store/`。

**职责：**
- 跨页面或跨模块共享状态的读写
- 派生数据（通过 selector / computed 模式）
- 跨域 action（委托给 API 层执行异步操作）

**设计约束：**
- 按业务域独立拆分，每个 store 文件对应一个业务域（如 `rectificationStore.ts`）
- Store 中的 action 可调用 API 层，但不依赖页面组件或业务视图
- 派生数据用 selector / computed 模式计算，不在 store 中冗余存储原始数据的加工结果
- 组件通过 Hook（如 `useStore`）访问 store，不直接 import store 实例操作状态
- 页面私有状态优先用组件本地 state 或页面级 Hook 管理，不下沉到 store

**Store 访问规则：**
- 页面编排层、业务视图层可直接通过 Hook **读取** Store 状态（如用户信息、门店信息）
- 对 Store 的**写操作**（dispatch action、修改状态）应通过业务逻辑层的 Hook 中转，保持写入路径可追溯

### 6. 共享基础层

放在 `src/components/`、`src/hooks/`、`src/types/`、`src/utils/`、`src/constants/`、`src/config/`。

**职责：**
- 通用 UI 组件（与具体业务弱绑定或无绑定）
- 跨页面复用 Hook
- 基础工具函数、常量、类型定义
- 基础设施能力（见下方横切关注点）

**横切关注点归属：**

| 能力 | 位置 | 说明 |
|------|------|------|
| 请求基础设施 | `src/utils/request.ts` | 封装 axios / fetch，统一拦截、错误处理、token 注入 |
| 路由配置 | `src/config/routes.tsx` | 路由表定义，不含业务逻辑 |
| 鉴权守卫 | `src/hooks/` 或 `src/components/` | 如 `usePermission`、`PermissionGuard` |
| 平台桥接 (H5) | `src/utils/bridge/` | Android / iOS / 小程序桥接适配 |
| 错误边界 | `src/components/ErrorBoundary.tsx` | 全局异常兜底 |
| 环境配置 | `src/config/env.ts` | 环境变量、feature flag |

**禁止：**
- 反向依赖 `pages/`
- 混入页面特定文案、接口参数或业务分支

## 目录分层建议

优先按"作用域"决定放置位置，而不是按个人习惯。

```text
src/
├── pages/                  # 页面入口与页面私有模块
│   └── patrol/
│       ├── index.tsx       # 页面编排层
│       ├── components/     # 业务视图层（页面私有组件）
│       ├── hooks/          # 业务逻辑层（页面私有 Hook）
│       ├── utils/          # 业务逻辑层（页面私有纯函数、Mapper）
│       └── api/            # 数据访问层（页面私有接口，可选）
├── api/                    # 数据访问层（全局共享接口）
├── store/                  # 状态共享层
├── components/             # 共享基础层 - 通用组件
├── hooks/                  # 共享基础层 - 通用 Hook
├── utils/                  # 共享基础层 - 工具函数、请求基础设施、桥接
├── types/                  # 共享基础层 - 业务类型 / 通用类型
├── constants/              # 共享基础层 - 常量
└── config/                 # 共享基础层 - 路由、环境配置
```

## 职责判定速查

遇到"代码该放哪里"时，优先按下面规则判断：

| 场景 | 建议位置 | 所属层 |
|------|----------|--------|
| 当前页面独有弹窗、卡片、列表片段 | 页面 `components/` | 业务视图层 |
| 当前页面独有请求编排、提交流程、草稿逻辑 | 页面 `hooks/` | 业务逻辑层 |
| 纯校验规则、得分计算、状态映射 | 页面 `utils/` | 业务逻辑层 |
| 后端字段 ↔ 前端模型的复杂业务转换 | 页面 `utils/xxxMapper.ts` | 业务逻辑层 |
| 接口级字段重命名、null 兜底 | API 函数内部 | 数据访问层 |
| 仅当前页面域使用的接口请求 | 页面 `api/` | 数据访问层 |
| 跨页面共用的接口请求 | `src/api/` | 数据访问层 |
| 跨页面共享状态 | `src/store/` | 状态共享层 |
| 跨页面共享但不依赖具体页面上下文 | `src/components/` / `src/hooks/` | 共享基础层 |
| 纯数据格式化、通用计算 | `src/utils/` | 共享基础层 |
| 仅某个页面临时使用的类型 | 页面内 `types.ts` 或同目录 | 就近放置 |
| 多处复用且语义稳定的类型 | `src/types/` | 共享基础层 |
| 请求拦截、token、错误处理 | `src/utils/request.ts` | 共享基础层 |
| 路由表、环境变量 | `src/config/` | 共享基础层 |

## 依赖方向

默认依赖方向：

```text
pages (页面编排层)
  ├──→ page components (业务视图层)
  │        ├──→ page hooks / utils (业务逻辑层)
  │        │        ├──→ page api / src/api (数据访问层)
  │        │        └──→ src/store (状态共享层，读写)
  │        └──→ src/store (状态共享层，只读)
  └──→ src/store (状态共享层，只读)

src/store → src/api / src/types
src/api → src/utils/request / src/types

src/components / src/hooks (共享基础层)
  → src/utils / src/types / src/constants
```

约束如下：

- 页面可以依赖共享层；共享层不能依赖具体页面。
- 业务视图层（组件）可以依赖业务逻辑层（Hook / utils）；反过来禁止——Hook 不 import 组件。
- 页面编排层和业务视图层可直接**读取** Store；对 Store 的**写操作**应通过业务逻辑层中转。
- 页面私有 Hook 之间可以组合调用，但避免互相缠绕形成隐式循环。
- `utils` 尽量保持纯函数，不依赖 React 运行时。
- `types` 不依赖业务实现文件。
- Store 可以调用 API 层，但不依赖页面或组件。
- 若出现双向 import，说明边界划分有问题，应重新拆分。

## 设计流程

### 1. 先识别变化轴

先判断本次需求主要变化落在哪一层：

- 只是页面结构 / UI 变化：优先留在页面编排层或业务视图层
- 是业务规则变化（校验、计算、状态转换）：优先落业务逻辑层
- 是接口差异或字段映射变化：优先落数据访问层
- 是多个页面共同诉求：再考虑下沉到共享层

### 2. 再定边界

为每个模块写清楚 3 件事：

- 它对外暴露什么
- 它依赖什么
- 它不应该负责什么

### 3. 最后再编码

编码前至少完成以下判断：

- 这段逻辑是否只服务当前页面？
- 如果未来复用，会不会带出页面上下文耦合？
- 抽出去后命名是否仍然语义清晰？
- 新增依赖是否会让调用链更长、更难测？

## 渐进式演进原则

- **先内聚后抽象**：先放在页面内收敛，复用稳定后再上提。
- **先接口稳定再共享**：不要把仍在频繁变化的模块过早做成全局能力。
- **优先新增模块，不污染旧模块**：新需求若与旧职责不同，应优先加新模块承接。
- **抽象要比复用晚一步**：看到 1 次相似代码不要立刻抽公共，至少确认模式稳定。
- **页面级 API 适时上提**：当页面级 `api/` 被第二个页面域引用时，迁移到 `src/api/`。

## 常见反模式

- `pages/` 中直接堆 500+ 行所有逻辑，页面既做流程又做细节实现。
- 全部逻辑都塞进一个 `useXxx`，导致 Hook 同时负责请求、校验、弹窗、导航、表格列定义。
- 把只在一个页面使用的代码提到 `src/components/` / `src/hooks/`，制造伪复用。
- API 函数直接返回未经整理的原始后端结构，导致上层到处写字段兼容代码。
- 公共组件中写大量业务 if/else，最后变成"万能组件"。
- 目录里出现 `common`、`helper`、`misc` 之类语义模糊的收纳桶。
- 共享模块反向 import 页面文件，形成隐式耦合。
- 在业务视图组件中内联大段校验 / 计算 / 数据转换，视图与逻辑纠缠。
- Store 中直接操作 DOM、弹 Toast 或依赖组件，Store 变成"万能控制器"。
- 数据适配逻辑散落在多个组件中，同一个字段映射在不同地方重复编写。

## 架构评审检查清单

- [ ] 页面层只做编排，不承载过多细节实现
- [ ] 业务视图层（组件）不内联大段业务计算或数据转换
- [ ] 业务逻辑层（Hook / utils）不反向依赖具体组件
- [ ] 页面私有逻辑未被错误上提到全局共享层
- [ ] 共享层未反向依赖页面层
- [ ] API 函数完成接口级数据整理，不向上层暴露原始后端结构
- [ ] 复杂业务转换（Mapper）集中在业务逻辑层，未在多处重复编写
- [ ] API 放置位置与复用范围匹配（页面级 vs 全局）
- [ ] Store 按业务域拆分，不含 UI 操作或组件依赖
- [ ] Store 读取路径简洁（页面/组件可直读），写入路径经业务逻辑层中转
- [ ] 依赖方向单向，无循环依赖
- [ ] 抽象粒度与复用范围匹配，未过度设计
- [ ] 新增模块命名清晰，目录位置符合作用域
- [ ] 横切关注点（请求、鉴权、路由、桥接）归属到共享基础层
- [ ] 架构调整具备渐进迁移与回滚空间

## 输出格式建议

涉及架构设计或分层调整时，交付内容建议包含：

1. 目标与约束（本次要解决什么，不改变什么）
2. 分层方案（模块清单、职责、依赖方向）
3. 目录落点（新增/修改文件放在哪里，为什么）
4. 演进路径（先收敛到哪里，何时再上提共享）
5. 风险与验证（是否存在循环依赖、伪复用、职责漂移）
