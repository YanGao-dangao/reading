# 迁移任务文档编写规范

两种文档类型，职责严格分离：
- **index.md**：阶段总览，扫一眼知全局，链接到各 task
- **task.md**：单任务操作手册，复制命令即可执行

---

## index.md 规范

### 必须包含的 6 个区块

```markdown
# Px 阶段名称

[← 返回总索引](../README.md)

**阶段目标**：一句话说清楚这个阶段要达成什么结果。
**对应目录**：明确列出本阶段涉及的目录和文件（如 `package.json`、`scripts/`）。
**完成情况**：⬜ 未开始 / 🟡 进行中 / ✅ 已完成

---

## 任务清单

（按功能模块分组的 checkbox 列表，提炼自各 task，供快速扫描）

- [ ] 模块A：做什么
  - [ ] 具体子项

---

## 验收标准

（只写可执行命令验证，不写主观描述）

- [ ] 执行 `npm run build:lib` 无报错
- [ ] `npx vue-tsc --noEmit` 通过

---

## Task 目录

| Task | 说明 | 预估工时 | 状态 |
|------|------|---------|------|
| [task-01 名称](./task/task-01-名称.md) | 一句话说明做什么 | 0.5h | ⬜ |
```

### index.md 三原则

1. **不写操作步骤**：操作细节属于 task.md，index.md 只写"做什么"，不写"怎么做"
2. **验收标准必须是命令**：`npm run xxx` / `node -e "..."` 可直接执行，不写"确保…正确"这类主观描述
3. **Task 目录表格完整**：每行必须有链接、说明、工时估算、状态

---

## task.md 规范

### 文件头元数据（必填）

```markdown
# Task-xx 任务名称

[← 返回 Px 索引](../index.md)

**状态**：⬜ 未开始
**优先级**：🔴 最高 / 🟠 高 / 🟡 中
**前置**：Task-xx（若有依赖，写清楚哪个 task 必须先完成）
**预估工时**：0.5h
**修改文件**：`path/to/file.ts`、`path/to/other.json`（执行前 reviewer 就能知道改哪里）
```

### 执行任务区块

每个子任务格式：

```markdown
### 子任务 N：任务名

说明背景或前置判断（一两句，非操作步骤）。

​```bash
# 注释说明这条命令的目的
具体可执行的命令
​```

关键判断点或注意事项（如兼容矩阵、备选方案）。
```

**命令编写规则**：
- 每条命令前有注释说明目的
- 涉及版本兼容性的，给出兼容矩阵表格
- 有备选方案的，写明触发条件（"若 X 不兼容，则…"）
- 临时调试代码必须标注 `（测试完删除）`

### 测试用例区块（核心，不可省略）

每个 TC 固定三段式：

```markdown
### TC-xx：测试目标描述

**操作**：
​```bash
可直接复制执行的命令
​```

**预期输出**：
​```
具体输出内容（不能写"正常输出"，必须写实际内容）
Vue version: 3.x.x
​```

**失败处理**：若出现 `具体报错信息`，执行 `具体修复命令`。
```

**TC 质量标准**：
- 操作必须是可执行命令，不是截图或 UI 操作描述
- 预期输出必须具体，写实际文本内容（版本号用 `x.x.x` 占位可接受）
- 失败处理必须有，不能留空；若确实无已知失败路径写"暂无已知失败路径"
- 高风险 task 需包含"反例验证"TC（故意破坏，验证检测机制有效）

### 修改文件区块

```markdown
## 修改文件

| 文件 | 操作 |
|------|------|
| `path/to/file.ts` | 新增 / 修改 / **预计无需修改** |
```

> 即使"预计无需修改"也要写出来，让 reviewer 知道你检查过了。

### 验收标准区块

```markdown
## 验收标准

- [ ] TC-01 通过：一句话描述通过条件
- [ ] TC-02 通过：一句话描述通过条件
- [ ] （其他不由 TC 覆盖的额外条件）
```

规则：
- 每条 checkbox 对应一个可验证结果
- 尽量引用 TC 编号，不引用时说明为何
- 不写模糊条件如"代码质量良好"

## 风险控制规则

### 必须写前置条件的情况
- 依赖其他 task 创建的文件
- 依赖特定工具版本（Node、npm、rollup 等）
- 修改共享配置（根 `package.json`、`tsconfig.json`）

### 必须提供备选方案的情况
- 版本兼容性不确定
- 网络请求（install 可能失败）
- 涉及破坏性修改（需要 rollback 路径）

### 原子性拆分原则
- 单个 task 修改文件 ≤ 5 个
- 单个 task 工时 ≤ 2h（超出则再拆）
- 每个 task 可以独立 review（不依赖同 PR 内其他 task 的代码改动）

---

## 文档元数据规范

```markdown
*最后更新：YYYY-MM-DD*
```

状态 emoji 含义统一：
- ⬜ 未开始
- 🟡 进行中  
- ✅ 已完成
- 🔴 阻塞中

---

## 快速检查清单

写完后逐项核对：

**index.md**
- [ ] 验收标准全部是可执行命令
- [ ] Task 表格有工时、有状态、有链接
- [ ] 没有操作步骤（步骤在 task.md 里）

**task.md**
- [ ] 文件头有：状态、优先级、前置、工时、修改文件
- [ ] 每个子任务有可复制命令
- [ ] 每个 TC 有：操作、预期输出、失败处理
- [ ] 高风险操作有备选方案或 rollback
- [ ] 修改文件表包含"预计无需修改"的文件
- [ ] 验收标准引用 TC 编号
