# React Guardrails

> **场景导航**
> - 新建页面或组件 → `/build-page`
> - 编码风格规范 → `/coding-standards`
> - 抽离方法、设计模式、SOLID → `/code-refactoring`
> - 架构分层设计 → `/architecture-design`
> - 清理无用代码 → `/dead-code-removal`

## 硬约束（必须满足）

- 不得改变既有业务流程、核心规则与页面主路径行为。
- 不得修改既有接口契约（请求参数、响应字段语义、状态码约定）除非用户明确授权。
- 不得改变既有权限、风控、计费、统计口径等关键业务语义。
- 所有改动默认向后兼容；若必须调整行为，必须通过显式开关或新入口隔离。
- 任何可能影响线上行为的改动，必须先给出影响点与回滚方案再实施。

## 适用场景

- 新建或改造 React 页面/组件
- 修复前端交互或状态相关问题
- 优化渲染性能、可维护性和可测试性
- 评审 React 代码质量并给出改进建议

## 默认技术取向

- 优先函数组件与 Hooks
- 优先 TypeScript 严格类型
- 优先“小而清晰”的组件边界
- 副作用集中在 `useEffect`/自定义 Hook，不散落在视图层

## 执行流程

1. **先理解上下文**
   - 明确页面目标、数据来源、交互路径、边界条件。
   - 识别已有约定：目录结构、状态方案、请求封装、UI 组件库。
   - 标记“业务不可变项”（流程、字段语义、权限、统计口径）。
2. **设计最小改动方案**
   - 优先复用已有组件与工具函数。
   - 先确定状态归属（本地状态 / 上层状态 / 全局状态）。
   - 明确兼容策略与回滚点。
3. **实现**
   - 先写类型与接口，再写组件逻辑与视图。
   - 将可复用逻辑抽成自定义 Hook（如 `useXxx`）。
   - 异步请求处理加载态、错误态、空态。
4. **自检**
   - 检查依赖数组、闭包陷阱、重复渲染、key 稳定性。
   - 验证可访问性基础项（语义标签、可聚焦控件、必要 `aria`）。
   - 对照硬约束逐条过一遍影响面。
5. **验证**
   - 跑 lint / 类型检查 / 测试（如项目已配置）。
   - 对关键业务路径做回归验证并记录结论。

## 实现准则

### 组件设计

- 每个组件只负责一个清晰职责。
- Props 语义化命名，避免 `data`, `obj`, `info` 这类泛名。
- 复杂组件拆分为：容器组件（数据/状态）+ 展示组件（纯 UI）。

### 状态管理

- 能用派生值就不用额外状态。
- 跨组件共享频繁时再上升状态或接入全局 store。
- 表单状态优先集中管理，避免多处互相覆盖。

### 副作用与请求

- `useEffect` 只放副作用，不做可同步计算。
- 请求需可取消或避免过期响应覆盖新状态。
- 严格区分：
  - 可恢复错误（toast + 保留页面）
  - 不可恢复错误（错误页/引导重试）

### 性能

- 先定位瓶颈，再做优化；避免“提前优化”。
- 对高频重渲染区域使用 `memo/useMemo/useCallback`，但保持可读性优先。
- 长列表使用虚拟列表或分页策略（视项目方案）。

### 质量与测试

- 关键交互至少覆盖：
  - 初始渲染
  - 用户操作
  - 请求成功/失败分支
- Bug 修复优先补回归测试（如果项目已有测试体系）。

## 快速检查清单

- [ ] 组件职责单一，命名清晰
- [ ] 状态没有冗余与循环依赖
- [ ] 异步逻辑有 loading/error/empty
- [ ] `useEffect` 依赖完整且无闭包陷阱
- [ ] 列表 key 稳定，不使用易变随机值
- [ ] 类型定义完整，边界值已处理
- [ ] 未改变既有接口契约与字段语义
- [ ] 未改变既有流程、权限、计费、统计口径
- [ ] 关键业务路径已完成回归验证

## 输出格式建议

在交付说明中优先包含：

1. 变更目标（一句话）
2. 关键实现点（2-4 条）
3. 新增/修改组件的介绍（职责、Props、使用场景）
4. 影响评估（明确说明“是否触及业务硬约束”）
5. 回滚方案
6. 验证步骤（如何本地复现与确认）
