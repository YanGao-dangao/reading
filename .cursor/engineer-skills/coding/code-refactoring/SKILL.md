# Code Refactoring

## 硬约束（必须满足）

- 不得改变既有业务规则、流程顺序、关键判断条件与异常分支语义。
- 不得改变对外契约（接口、事件、消息结构、持久化字段、请求/响应语义）除非用户明确授权。
- 不得影响既有权限、风控、计费、统计口径与审计链路。
- 重构默认等价迁移：先保证行为一致，再谈结构优化。
- 任何行为差异必须显式隔离（开关/新入口），并提供回滚路径。

## 通用执行流程

1. **理解上下文**：明确目标、不可变项、对外契约边界。
2. **设计最小改动**：优先复用，明确兼容策略与回滚点。
3. **小步迁移**：每次改动后验证行为等价，不跨越式重写。
4. **验证收敛**：关键路径回归，清理临时桥接与死代码。

### 1. 设计模式

- **适用**：耦合高、分支膨胀、同类逻辑重复、新需求导致修改范围过大。
- **速查**：行为切换 → 策略；流程骨架稳定 → 模板方法；状态驱动 → 状态模式；事件广播 → 观察者；创建复杂 → 工厂；接口不兼容 → 适配器；统一子系统 → 外观；可撤销 → 命令。
- **原则**：先识别问题再选模式；最小重构路径；复杂度可控。
- **详细**：`references/reference.md`（锚点：设计模式）

### 2. 抽离方法

- **适用**：函数过长（>40-60 行）、重复逻辑块、复杂条件分支、需降低圈复杂度。
- **不适用**：5-10 行直观函数；抽离后参数爆炸；业务边界尚不稳定。
- **准则**：一次只抽一个清晰意图；方法名表达“做什么”；参数 ≤4-5 个，否则考虑对象参数。
- **详细**：`references/reference.md`（锚点：抽离方法）

### 3. SOLID 原则

- **适用**：职责混杂、接口臃肿、高层依赖底层、新增需求只能改老逻辑。
- **速查**：SRP 一个变化原因；OCP 扩展开放修改关闭；LSP 子类型可替换；ISP 小接口；DIP 依赖抽象。
- **详细**：`references/reference.md`（锚点：SOLID 原则）

## 通用检查清单

- [ ] 改动基于真实痛点，非主观偏好
- [ ] 依赖方向清晰，未引入循环依赖
- [ ] 迁移可回滚，关键路径有验证
- [ ] 未改变既有对外契约与业务语义
- [ ] 关键业务路径已完成等价性回归

## 输出格式建议

交付时建议包含：

1. 变更目标与影响范围
2. 关键实现点或选用方案及理由
3. 迁移步骤与回滚方案
4. 等价性验证结果（明确说明“未影响既有业务逻辑”）
