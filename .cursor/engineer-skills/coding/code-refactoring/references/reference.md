# Code Guardrails 详细细则

本文档为 SKILL.md 各专题的详细说明，按需查阅。

## 设计模式

### 适用场景

- 代码耦合高、分支膨胀、难扩展
- 同类逻辑重复，缺乏统一抽象
- 新需求频繁导致修改范围过大
- 需要在重构中保持行为兼容

### 使用原则

- 先识别问题，再选模式；不要“为模式而模式”。
- 优先最小重构路径，保证可回滚。
- 在当前团队认知范围内选择复杂度可控的方案。

### 模式选择速查

按“问题类型”选：

| 问题类型 | 模式 |
|----------|------|
| 行为切换频繁 | 策略模式（Strategy） |
| 流程骨架稳定、步骤可变 | 模板方法（Template Method） |
| 状态驱动行为变化明显 | 状态模式（State） |
| 事件广播或跨模块通知 | 观察者模式（Observer） |
| 创建逻辑复杂或需要解耦构造过程 | 工厂模式（Factory） |
| 外部系统接口不兼容 | 适配器模式（Adapter） |
| 需要统一复杂子系统调用 | 外观模式（Facade） |
| 历史操作可撤销/回放 | 命令模式（Command） |

### 执行流程

1. **定位异味**：长函数、超大类、重复分支、条件爆炸、跨层依赖；标记不可变业务规则与对外契约点。
2. **定义目标**：明确要提升的指标（可扩展性、可测试性、可读性或稳定性）。
3. **选择模式并画边界**：定义参与角色；约束依赖方向，避免新一轮耦合。
4. **小步重构**：先抽接口，再迁移一条主路径，最后补齐其余分支；每一步保持功能可运行并验证业务行为等价。
5. **验证与收敛**：对比重构前后行为；清理临时桥接代码与死代码。

### 抽象层设计

- 接口名表达“能力”而非“实现细节”。
- 抽象层只暴露必要行为，避免“胖接口”。

### 依赖管理

- 依赖反转：高层不直接依赖具体实现。
- 使用组合优先于继承（除非继承关系稳定且语义清晰）。

### 兼容迁移

- 旧入口保留过渡期，内部委托到新实现。
- 通过 feature flag 或配置开关控制新旧路径切换（项目支持时）。

### 可测试性

- 每个模式角色至少有一个行为测试。
- 优先测试“协作契约”而非私有实现细节。
- 对关键业务路径补等价性回归测试。

### 反模式提醒

- 用单例隐藏全局状态，导致测试困难。
- 模式层级过多，业务逻辑被“框架化”掩盖。
- 抽象过早，接口长期只有一个实现却复杂度显著增加。

### 检查清单

- [ ] 选择模式基于真实痛点，而非主观偏好
- [ ] 新增抽象确实减少了分支/耦合
- [ ] 依赖方向清晰，未引入循环依赖
- [ ] 迁移过程可回滚，关键路径有验证
- [ ] 文档说明了模式角色与扩展点
- [ ] 重构前后关键业务行为等价
- [ ] 未改变既有对外契约与业务语义

### 何时不使用设计模式

- 问题规模很小且变化概率低
- 直接函数拆分即可解决可读性问题
- 团队当前阶段更需要一致性而非抽象能力

## 抽离方法

### 适用场景

- 单个函数过长（通常 > 40-60 行）且包含多个语义片段
- 同一函数内出现重复逻辑块，适合提炼复用
- 复杂条件分支导致阅读和定位问题困难
- 需要降低圈复杂度，提升单元测试可测性

### 不适用场景

- 仅 5-10 行且语义非常直观的函数
- 抽离后会引入大量参数透传，反而增加理解成本
- 业务仍在高频变动、边界尚不稳定（先明确规则再抽离）

### 方法抽离决策准则

- 只抽“一个清晰意图”的代码块；不要把多个意图混在一个新方法。
- 抽离后方法名应能表达“做什么”，而不是“怎么做”。
- 优先减少临时变量跨作用域传播，避免新方法参数爆炸。
- 如果抽离后参数超过 4-5 个，优先评估是否应先引入对象参数或上移状态归属。

### 执行流程

1. **锁定边界**：标记不可变行为（输入校验、关键分支、错误处理、返回值格式）；明确前置条件与副作用（状态写入、网络调用、日志、埋点）。
2. **识别候选代码块**：优先抽离重复块、深层嵌套块、注释才能看懂的块；保留主流程编排、顺序敏感操作。
3. **设计签名**：入参只传最小必要数据，返回值聚焦单一结果；明确错误策略（抛错/返回空值/Result），保持与原逻辑一致。
4. **小步迁移**：一次只抽一个方法；每次改动后立即验证编译/测试/关键路径；先抽纯逻辑，再抽带副作用逻辑。
5. **验证收敛**：对比重构前后输出、异常、日志/埋点；清理无用变量与重复分支。

### 命名规范

- 方法名优先“动词 + 业务对象”：`validateUploadFile()`、`buildMultipartParams()`、`mergeUploadedParts()`
- 布尔返回方法用 `is/has/can/should` 前缀：`isFileTypeAllowed()`、`shouldUseMultipartUpload()`

### 参数与返回设计

- 入参优先不可变数据，避免在子方法内部隐式修改外部状态。
- 返回值优先显式结构，减少通过闭包写外部变量。
- 若必须修改外部状态，在方法注释中明确副作用。

### 常见反模式

- 机械拆分：每 5 行抽一个方法，导致调用链碎片化。
- 过度透传：将大量上下文参数层层传递。
- 抽离后丢失异常上下文，错误信息变得不可定位。
- 把时序敏感逻辑拆散，改变执行先后顺序。

### 质量门禁（每次抽离后检查）

- [ ] 主流程可读性明显提升（读主函数无需跳转太多）
- [ ] 新方法职责单一，命名清晰
- [ ] 入参/返回值与副作用边界清楚
- [ ] 关键业务路径行为与重构前一致
- [ ] 编译、lint、测试或最小回归通过
- [ ] 回滚路径明确（可快速恢复到抽离前）

## SOLID 原则

### 适用场景

- 类/模块职责混杂，改动一个需求牵连多个方向
- 接口臃肿，调用方被迫依赖不需要的方法
- 高层业务直接依赖底层实现，替换成本高
- “新增需求只能改老逻辑分支”，扩展风险高

### 原则速查（SOLID）

| 原则 | 含义 |
|------|------|
| **S: 单一职责（SRP）** | 一个模块只承担一个变化原因 |
| **O: 开闭原则（OCP）** | 对扩展开放，对修改关闭；新增能力优先新增实现而非修改稳定主干 |
| **L: 里氏替换（LSP）** | 子类型可替换父类型且不破坏调用方预期行为 |
| **I: 接口隔离（ISP）** | 使用多个小接口，避免“胖接口”强迫依赖 |
| **D: 依赖倒置（DIP）** | 高层依赖抽象，不依赖具体实现 |

### 执行流程

1. **定位职责问题**：标记“一个模块做多件事”“一改多处崩”的热点区域；明确不可变业务语义与对外契约边界。
2. **定义重构目标**：目标聚焦可扩展、可测试、可替换、可读性提升。
3. **按原则拆解**：SRP 按变化原因拆分；OCP 将变化点抽象成扩展点；ISP 拆分大接口；DIP 引入抽象层。
4. **小步迁移**：先保留旧入口，内部委托新实现；逐步替换调用方；每一步验证行为等价。
5. **验证收敛**：对关键业务路径做回归；清理临时适配层与死代码。

### 单一职责（SRP）落地

- 按“变化原因”而非“技术层”拆分。
- 避免一个模块同时处理：校验、编排、持久化、展示等多种职责。
- 若函数超过团队可维护阈值（分支复杂、上下文过多），优先做职责拆分。

### 开闭原则（OCP）落地

- 对已稳定主流程，新增需求优先通过新实现接入，不直接改主干分支。
- 在扩展点定义稳定契约，扩展实现只关注差异行为。

### 里氏替换（LSP）落地

- 子实现不得削弱父契约（输入前置条件更严格、输出承诺更弱都要避免）。
- 调用方不应因替换实现而修改业务判断。

### 接口隔离（ISP）落地

- 按调用方分组接口能力，避免“全量能力接口”。
- 接口命名表达能力边界，避免语义泛化。

### 依赖倒置（DIP）落地

- 业务编排层依赖抽象接口，不直接 new 具体实现。
- 通过注入/工厂组装依赖，降低耦合与测试成本。

### 反模式提醒

- 以“分层”替代“职责”：形式上分层，实质仍职责混杂。
- 抽象过度：只有一个实现却提前引入复杂抽象体系。
- 接口过大：为图省事把不相关能力塞进同一接口。

### 检查清单

- [ ] 每个模块只有一个主要变化原因（SRP）
- [ ] 新需求通过扩展点接入，而非频繁改主干（OCP）
- [ ] 替换实现后调用方无需改业务判断（LSP）
- [ ] 调用方只依赖其所需最小接口（ISP）
- [ ] 高层仅依赖抽象，不依赖具体实现（DIP）
- [ ] 未改变既有业务语义与对外契约
- [ ] 关键业务路径已完成等价性回归验证
