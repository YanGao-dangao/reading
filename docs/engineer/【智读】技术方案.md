## 2. 范围与边界

- **P0 严格范围**：仅实现 F01-F25。
- **P1 排除项**：F26-F29（文件上传、用户体系高级能力）本期不做业务交付，仅保留可扩展接口位。
- **后端与数据库**：本期必须后端落库，不采用纯前端本地存储方案。
- **流式协议**：内容生成统一使用 `SSE`。
- **多用户隔离**：首期即按多租户/多用户隔离建模。
- **历史回看**：仅支持历史列表查看 + 恢复，不做版本差异对比。

## 3. 技术选型（已确认）

### 3.1 决策表

| 决策点 | 备选 | 最终选型 | 说明 |
|---|---|---|---|
| 前端框架 | React + Antd / 其他组件库 | **React + Antd** | 满足后台/业务界面快速交付 |
| 后端形态 | bun-server / Node 框架 | **bun-server** | 你指定 `bun-server` 作为服务框架 |
| 数据库 | PostgreSQL / MySQL | **PostgreSQL** | JSONB、全文检索扩展、事务能力更适配生成场景 |
| ORM | Prisma / Drizzle | **Prisma** | 迁移、类型安全、团队上手成本低 |
| 搜索 Provider | Tavily / SerpAPI / 可插拔抽象 | **Provider 抽象 + Tavily 首实现** | 你已确认“都可以”，建议保留切换能力 |
| LLM Provider | DeepSeek（主）+ 可扩展 | **DeepSeek + 抽象层** | 满足当前偏好并预留切换 |
| 流式输出 | SSE / WebSocket | **SSE（已确认）** | 服务器单向推送，前端实现简单 |
| 鉴权 | JWT + HttpOnly Cookie / Session | **JWT + HttpOnly Cookie** | 便于多端接入，首期实现成本可控 |
| 多租户策略 | 逻辑隔离（tenant_id）/ 物理隔离 | **逻辑隔离** | 首期成本最低，后续可按租户分库演进 |
| 运行时与包管理 | bun / node + npm | **bun** | 统一前后端构建与依赖管理体验 |

### 3.2 Gate 2 结论（已确认）

1. 技术栈采用：`React + Antd + bun-server + PostgreSQL + bun + Prisma`。
2. 搜索采用：`Provider 抽象 + Tavily 首实现`。
3. 鉴权采用：`JWT + HttpOnly Cookie`，并执行全链路 `tenant_id` 隔离。

## 4. 整体架构

### 4.1 架构概览

```text
Browser (Web)
   │
   │ HTTPS
   ▼
React Web App (Antd)
   │
   │ HTTPS
   ▼
bun-server API
   ├── Search Provider Layer (Tavily / SerpAPI Adapter)
   ├── LLM Provider Layer (DeepSeek Adapter)
   ├── Generation Service (Outline / Brief / Detailed / Insights)
   ├── SSE Stream Gateway (/api/generation/stream)
   └── Persistence Service (Prisma)
            │
            ▼
        PostgreSQL
```

### 4.2 模块划分

| 模块 | 职责 | 涉及功能 |
|---|---|---|
| Search Module | 书名搜索、候选结果、无结果降级触发 | F01-F03 |
| Outline Module | 大纲生成、树结构存储与加载 | F04-F06 |
| Generation Module | 缩略/详细/心得生成、多章节合并、重生成 | F07-F17 |
| Interaction Module | 复制、加载态、超时与错误反馈 | F18-F20 |
| Persistence Module | 书籍历史、生成历史、恢复 | F21-F23 |
| Responsive UI Module | 桌面双栏与移动抽屉交互 | F24-F25 |
| Tenant/Auth Module | 租户与用户隔离、权限边界 | 支撑全局 |

### 5.2 关键接口：创建任务 + SSE

**API-READ-005 请求**

```json
{
  "bookId": "uuid",
  "chapterIds": ["uuid-1", "uuid-2"],
  "mode": "brief",
  "params": {
    "style": "通俗易懂",
    "wordCount": 300
  },
  "regenerateFromId": "optional-generation-id"
}
```

**API-READ-005 响应**

```json
{
  "jobId": "uuid",
  "streamUrl": "/api/read/generation/stream?jobId=uuid",
  "timeoutSec": 30
}
```

**API-READ-006 SSE 事件**

```text
event: token
data: {"jobId":"...","chunk":"本段新增文本"}

event: done
data: {"jobId":"...","generationId":"...","usage":{"input":1234,"output":567}}

event: error
data: {"jobId":"...","code":"TIMEOUT","message":"生成超时，请重试"}
```

## 6. 数据模型（首期多用户隔离）

## 6.1 实体清单

| 表/实体 | 关键字段 | 说明 |
|---|---|---|
| `tenants` | `id`, `name`, `status` | 租户 |
| `users` | `id`, `email`, `display_name`, `status` | 用户 |
| `tenant_users` | `tenant_id`, `user_id`, `role` | 用户-租户关系 |
| `books` | `id`, `tenant_id`, `title`, `author`, `source_type`, `source_meta` | 书籍主数据 |
| `book_outlines` | `id`, `tenant_id`, `book_id`, `version`, `outline_json` | 大纲存储（JSON 树） |
| `chapters` | `id`, `tenant_id`, `book_id`, `parent_id`, `title`, `order_no` | 可选：章节索引（检索优化） |
| `generation_jobs` | `id`, `tenant_id`, `user_id`, `book_id`, `mode`, `status`, `params_json` | 生成任务 |
| `generations` | `id`, `tenant_id`, `user_id`, `book_id`, `job_id`, `content_md`, `chapter_ids_json` | 生成结果 |
| `search_histories` | `id`, `tenant_id`, `user_id`, `query`, `selected_book_id` | 搜索历史 |

## 6.2 多租户隔离规则

- 所有业务查询必须带 `tenant_id` 条件。
- `tenant_id + user_id` 作为历史与生成数据的访问边界。
- 管理端能力（P1）再扩展角色权限；P0 仅保证数据隔离正确。

## 8. 非功能设计

| 关注点 | 方案 |
|---|---|
| 性能 | 搜索 P95 < 3s；首 token 返回 < 2s（命中模型可用时） |
| 稳定性 | 生成任务状态机（queued/running/success/failed/timeout） |
| 安全 | 租户级数据隔离；接口鉴权；敏感日志脱敏 |
| 可观测 | 请求日志、LLM 调用日志、SSE 断流统计、错误码聚合 |

## 9. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 第三方搜索波动 | 搜索结果不稳定 | Provider 抽象 + 熔断 + 降级到 AI 概要 |
| LLM 超时/限流 | 生成失败率上升 | 重试策略、超时事件、用户可重生成 |
| 多租户条件漏加 | 数据越权 | 数据访问层统一注入 `tenant_id`，加集成测试 |
| SSE 连接中断 | 用户体验下降 | 前端自动重连 + 任务状态轮询兜底 |
